# 基准测试协议

## 概述

OpenProblemAtlas 可以作为在真实开放问题上评估 AI 推理系统的基准。本文档描述了构建公平的、具有污染感知能力的评估子集的协议。

## 原则

1. **Atlas 优先，基准其次。** 首要目标是建立可靠的数据库。基准测试用途是下游应用。
2. **滚动子集。** 评估子集从主 Atlas 中抽取，按月冻结。新鲜问题降低污染风险。
3. **禁止评测注水。** 部分结果或启发式解法不算作"已解决"。仅已验证的状态变更才算数。
4. **可复现性。** 每次评估必须记录：模型、版本、提示词、工具、日期以及使用的精确快照。

## 每月快照流程

1. 每月1日，从当前规范数据生成冻结快照。
2. 快照以 GitHub 发布形式打标签：`snapshot-YYYY-MM-DD`。
3. 快照包含：
   - `problems.json` -- 所有已验证问题
   - `problems.parquet` -- 相同数据的 Parquet 格式（如可用）
   - `stats.json` -- 汇总统计
   - `changelog.md` -- 自上次快照以来的变更

## 评估子集构建

### 滚动子集

为评估选取的问题子集，按月刷新：

- **规模：** 50-100 个问题
- **选取标准：**
  - 状态：仅 `open`
  - 置信度：`high`
  - 形式化分数：>= 0.7
  - 具有规范描述
  - 3 个月内未出现在任何之前的滚动子集中（防污染）

### 污染感知

- 出现在流行训练数据集（如 MATH、GSM8K、竞赛存档）中的问题会被标记。
- `contamination_risk` 字段（未来将添加）将指示：
  - `low` -- 问题小众，不太可能出现在训练数据中
  - `medium` -- 问题知名，但描述是原创的
  - `high` -- 问题和解法在网上广泛可得

### 难度层级

| 层级 | 描述 | 专家典型用时 |
|------|------|------------|
| 1 | 研究级别，无已知解法 | 数小时到数年 |
| 2 | 已知困难，存在部分结果 | 数小时到数天 |
| 3 | 使用工具可接近，是否开放不确定 | 数分钟到数小时 |

## 评估指标

用于评估尝试开放问题的 AI 系统：

| 指标 | 描述 |
|------|------|
| `progress_rate` | 取得有意义部分进展的问题比例 |
| `decomposition_quality` | 问题分解的质量（专家评分） |
| `formalization_rate` | 具有有效形式化描述的问题比例 |
| `false_claim_rate` | 包含错误声明的尝试比例（越低越好） |
| `reproducibility` | 可复现的尝试比例 |

## 报告

基准测试结果应包含：
- 使用的快照版本
- 尝试的问题 ID 完整列表
- 逐问题的尝试记录（遵循尝试模式）
- 汇总指标
- 已知局限性

## 防作弊

- 不要在模型训练数据可能包含解法的问题上进行评估。
- 未经独立验证，不要声称"已解决"。
- 报告负面结果和失败 -- 它们是有价值的数据。
