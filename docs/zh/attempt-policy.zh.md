# 尝试策略

## 什么是尝试？

尝试是指由 AI 系统、人类或人机混合体对已验证的开放问题所做的记录性工作。尝试存储在 Lab 层中，不会污染 Atlas 真实数据层。

## 必填字段

每个尝试必须包含以下字段：

| 字段 | 描述 |
|------|------|
| `problem_id` | 引用 Atlas 中的已验证问题 |
| `actor` | 谁/什么进行了尝试 |
| `actor_type` | `human`、`ai` 或 `hybrid` |
| `model` | 模型名称和版本（用于 AI 尝试） |
| `date` | 尝试的日期 |
| `method_summary` | 使用了什么方法 |
| `tools` | 使用的软件、库、硬件 |
| `outcome` | 结果是什么 |
| `verification_status` | `verified` 或 `unverified` |
| `limitations` | 尝试的已知局限性 |

## 默认标签

所有新尝试会被标记为：
- `unverified` -- 直到被独立确认
- `machine-generated` 或 `hybrid` -- 如果涉及 AI
- `not canonical` -- 尝试结果不是规范事实

## 什么算作尝试

- LLM 文献综述或问题分解
- 计算搜索（暴力搜索、SAT、符号计算）
- Lean/Coq 形式化尝试
- 反例搜索
- 证明草图或部分证明
- 包含分析的可复现笔记本

## 什么不算尝试

- 没有任何执行的模糊"我觉得这可能行"
- 单次 LLM 提示且没有对输出的分析
- 没有证据的营销宣称（如"GPT-5 解决了这个！"）

## 验证

尝试可以在满足以下条件时从 `unverified` 升级为 `verified`：
1. 方法可复现
2. 独立审核者确认了结果
3. 对于证明声明：领域专家或形式化验证确认

## 展示规则

在问题旁展示尝试时：
- 始终展示：执行者、模型、版本、日期、工具、验证状态
- 始终展示局限性
- 绝不将未验证的 AI 输出呈现为事实
- 排序方式：按日期（最新优先），然后按验证状态

## AI 尝试的成本管理

- 使用小型模型进行初步筛选
- 将大型模型留给入围问题
- 缓存所有输出
- 记录提示词版本和模型 ID
- 确保所有机器生成的字段可追溯
